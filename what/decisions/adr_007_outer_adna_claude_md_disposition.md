# ADR-007: Outer `adna/CLAUDE.md` becomes `how/templates/template_workspace_claude.md`

## Status

Proposed (Stage 2 Session 1 of `campaign_adna_v2_infrastructure`, drafted 2026-05-08 by Rosetta as M01 Obj 2 deliverable). **Operator ratifies to `accepted` at the start of M03 execution per Standing Order #1.** Companion to [[adr_006_github_repo_rename_to_adna.md|ADR-006]].

## Context

The current adna template repo has a two-level structure:
```
adna/                                ← git repo
├── CLAUDE.md                        ← OUTER WRAPPER (5805 bytes; "Do not edit this file")
└── .adna/                           ← TEMPLATE CONTENT
    └── CLAUDE.md                    ← TEMPLATE GOVERNANCE v6.0 (~21 KB; the "real" CLAUDE.md)
```

The M03 repo flatten ([[../../how/campaigns/campaign_adna_v2_infrastructure/missions/mission_adna_infra_planning_01.md|mission spec]] §Obj 2; consequence of the structural simplification) removes the outer layer:
```
.adna/                               ← IS the git repo
└── CLAUDE.md                        ← only one CLAUDE.md remains (template governance, formerly .adna/CLAUDE.md)
```

The flatten leaves the **outer `adna/CLAUDE.md`** with no destination. M01 Stage 2 Session 1's [[../../how/campaigns/campaign_adna_v2_infrastructure/missions/artifacts/m01_obj1_current_state.md|Obj 1 deliverable]] §1 captured the file's content and §5 Issue **I-8** flagged it as the disposition decision this ADR resolves.

### Audit of outer `adna/CLAUDE.md` content (per Obj 1 Step 2)

| Section | Lines | Unique to this file? | Equivalent elsewhere |
|---|---|---|---|
| `## Identity` (Berthier intro) | 22–37 | **Yes** — workspace-chief-of-staff framing distinct from `.adna/CLAUDE.md`'s template-governance Berthier voice | None — this is workspace-router persona, not template-governance persona |
| `## Startup: State Detection` (Steps 1–3) | 38–63 | **Yes** — three-step bootstrap detection for a workspace-root agent (verify template + scan projects + route by state) | `.adna/CLAUDE.md` has its own First-Run Detection but for a forked-project context, not a workspace-router context |
| `## Project Creation` | 64–76 | Partial — delegates to `skill_project_fork.md` which is canonical | Skill is canonical; this section is a thin wrapper |
| `## Tool Check` | 78–86 | **Yes** — git/python3/Obsidian environment-check guidance | None |
| `## Standing Rules` (4 rules) | 87–94 | Partial — overlaps with `.adna/CLAUDE.md` standing orders + `lattice/CLAUDE.md` Standing Rules | Other files have rules that say similar things but in different operational contexts |
| `## Compute Tiers` | 96–105 | **No** — duplicates `.adna/CLAUDE.md` Compute Tiers section verbatim | `.adna/CLAUDE.md` lines 295–303 |
| `## For Full Documentation` (link table) | 107–117 | Partial — lists where to find everything inside `.adna/` | Useful as a navigation index |

**Summary**: ~60 lines of unique content (Identity / Startup / Tool Check / link table) + ~20 lines of governance overlap + ~15 lines of full duplication. The unique content describes **workspace-router behavior** that `~/lattice/CLAUDE.md` exhibits for a real operator's machine.

### Why a separate file is needed (not just merge into `.adna/CLAUDE.md`)

`.adna/CLAUDE.md` (template governance) governs **a forked project's agent behavior** (the `agent_init` first-run, the campaign/mission/session protocol inside *that* project). It is what every `.aDNA/` project's CLAUDE.md is expected to look like after onboarding.

The outer wrapper governs **the workspace agent** (the agent operating at `~/lattice/`, before any project exists or to route across projects). These are different contexts. Conflating them would force `.adna/CLAUDE.md` to either pick one stance (workspace-only or project-only) or carry both — which would bloat it and confuse first-time readers.

### Why a docs file is sub-optimal (Option 3 from mission §Obj 2 §Decision B)

A docs file at `.adna/how/docs/workspace_setup_guide.md` would document the workspace-bootstrap pattern in prose. It would be useful as supplementary reading. But it would not be **directly installable** — operators reading the docs would still need to author `~/lattice/CLAUDE.md` from scratch. The outer wrapper today is *almost* a workspace-router CLAUDE.md; converting it to a template makes it directly installable.

### Why a template file is the right form (Option 4 — chosen)

The aDNA pattern for "content meant to be copied to a destination" is the **template** (`how/templates/template_*.md`). The v5.7 CHANGELOG entry already mentions a `workspace_claude_md.template` ("workspace_claude_md.template updated with fork preparation steps") — but **no such file currently exists** in `.adna/how/templates/` (verified 2026-05-08). The slot is documented but vacant. ADR-007 fills it.

## Decision

**Convert the outer `adna/CLAUDE.md` to `.adna/how/templates/template_workspace_claude.md` during the M03 flatten.**

The conversion is content-preserving with three minor edits:

1. **Frontmatter**: replace the current `type: governance` frontmatter with the standard template frontmatter (per other `template_*.md` files):
   ```yaml
   ---
   type: template
   template_for: workspace_claude_md
   created: 2026-04-03
   updated: <M03 date>
   status: active
   last_edited_by: agent_stanley
   tags: [template, workspace, claude_md, lattice, governance]
   ---
   ```
2. **Title**: change `# CLAUDE.md — Your Lattice` to `# Template — Workspace CLAUDE.md` (template-naming convention) and add a short preamble explaining "*Use this as the starting point for `~/lattice/CLAUDE.md` — your workspace router. Customize Project Discovery and Workspace Layout sections to your installed projects.*"
3. **"Do not edit this file" disclaimer** (line 13 of current outer): remove. As a template, this file IS meant to be copied (and the copy customized). Add an inline comment instead: `<!-- This is a template. Copy to ~/lattice/CLAUDE.md and customize. Updates flow via git pull on the .adna/ clone. -->`

**Sections retained as-is**: Identity (Berthier), Startup State Detection, Project Creation, Tool Check, Standing Rules, Compute Tiers, For Full Documentation.

**Sections optionally trimmed during M03 (Obj 7 simplification opportunity)**: Compute Tiers if `.adna/CLAUDE.md` already has it (replace with `<!-- See .adna/CLAUDE.md §Compute Tiers -->`); the link table at the end if `.adna/README.md` already has it.

**Workflow integration**:
- `skill_workspace_upgrade.md` (M03 update target) gains a step: *"Step 3 alternative — install workspace router from template"*: copy `.adna/how/templates/template_workspace_claude.md` → `~/lattice/CLAUDE.md` if no workspace router exists.
- `skill_workspace_init.md` was deprecated in v6.0 (per `.adna/CLAUDE.md` line 221) and is being formally retired in M03. The workspace-init flow is now covered by manual-onboarding-via-template plus `skill_workspace_upgrade.md` for workspaces with existing content.
- `skill_onboarding.md` (project-fork onboarding flow) is unaffected.

## Consequences

### Positive

- **Self-consistent template form**: workspace-router content lives where templates live (`.adna/how/templates/`) — same convention as every other piece of meant-to-be-copied content.
- **Fills the v5.7-documented but missing template slot** (Obj 1 §3 templates audit identified the gap implicitly; this resolves it).
- **Cleaner repo root post-flatten**: `.adna/` no longer carries a wrapper `CLAUDE.md` whose role is ambiguous; only the template-governance CLAUDE.md remains at `.adna/CLAUDE.md`.
- **Documented bootstrap path for new operators**: the gap between "I cloned `.adna/`" and "I have a working `~/lattice/CLAUDE.md`" now has an explicit, copyable artifact.
- **Workspace-router-specific Berthier persona is preserved** — distinct from project-Berthier in `.adna/CLAUDE.md`, distinct from project-specific personas (Rosetta, Daedalus, …) in actual project vaults.

### Negative

- **One-time skill update**: `skill_workspace_upgrade.md` needs a Step 3 alternative path (template-based bootstrap). M03 absorbs this work.
- **Existing workspaces (notably Stanley's `~/lattice/CLAUDE.md`)** were authored without this template — the template inherits *from* Stanley's site-specific router rather than the other way around. Mitigation: the template can be backported (if useful) by extracting the structural content of Stanley's router and stripping site-specific bits. M03 is **not** required to do this backport; the template can be authored fresh from the outer wrapper as documented above.
- **Discoverability of the template**: operators won't find it unless `skill_workspace_upgrade.md` or `.adna/CLAUDE.md` references it. Mitigation: M03 ensures both files reference the new template path.

### Neutral

- **`.adna/CLAUDE.md` is unchanged** by this ADR — its role (template governance for forked projects) stays exactly the same.
- **The site-specific `~/lattice/CLAUDE.md`** is also unchanged for existing workspaces — operators only adopt the template if they're spinning up a new workspace or want to refresh their router.
- **The outer wrapper's "Do not edit this file" rule** stops applying to the new template file because templates ARE meant to be copied and customized — but the original "do not edit" rule for the **template file in the repo** transfers via the template frontmatter pattern (`status: active` + the in-template comment about updates flowing via git pull).

## Alternatives considered

| Option | Rationale to reject (vs. chosen Option 4) |
|---|---|
| **Option 1 — Merge into `.adna/CLAUDE.md`** | Conflates two distinct agent contexts (workspace-router vs. forked-project-governance). `.adna/CLAUDE.md` is already 21 KB and well-scoped; adding workspace-router behavior would bloat and confuse it. |
| **Option 2 — Drop entirely** | Loses ~60 lines of unique workspace-router content (Berthier identity, Startup State Detection, Tool Check, link table). Operators bootstrapping a fresh `~/lattice/` would have nothing to base their workspace router on except whatever they author from scratch. |
| **Option 3 — Convert to `.adna/how/docs/workspace_setup_guide.md`** | Useful for prose explanation but not directly copyable. New operators would still need to translate docs into a working CLAUDE.md by hand. The template form (Option 4) makes the artifact directly installable. *However*: a shorter `workspace_setup_guide.md` may still be valuable as a *companion* to the template; M07 may add one. |
| **Option 4 — Convert to `template_workspace_claude.md` (CHOSEN)** | Salvages content as a directly-installable template; aligns with the existing `template_*.md` pattern; fills the v5.7-documented but missing slot; preserves the workspace-router Berthier persona; M03 cost is bounded (one file move + 3 minor edits + skill_workspace_upgrade integration). |
