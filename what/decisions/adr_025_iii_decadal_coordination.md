Without a canonical doctrinal answer, every consumer wrapper + future M3.7 implementation phase + III.aDNA Campaign E (or successor) would re-derive the cycle-location-shape ad hoc.

**Three motivations for in-phase ADR-025 ratification** (per Campaign SO #14 in-phase exception clause):

3. **Load-bearing for cross-vault III-decadal coord memos**: ADR-025 Clause C ack-debt-protocol enables coord-memo-based cycle scheduling across vaults without phase-exit-gating each cross-vault dispatch. M3.4 + M3.5 + M3.6 + M3.7 all cite the same trigger (load-bearing for in-phase consumers + cross-vault propagation that would otherwise stall on phase-exit gating) — the clause variant is now self-perpetuating doctrine.

## Decision

### Clause A — Cycle-location-shape: distributed-but-coordinated

**III-decadal cycles RUN at multiple vault locations under a single coordination contract**. Specifically:

- **Operator-side cycles RUN at aDNA.aDNA** — measure operator-vault-context dimensions (`persona_growth_v0` + `research_context_generation_v0` measure persona blurb drift + adopter-scenario research-context generation against the operator's vault collection). Operator-side cycles are dispatched via `skill_iii_modular_cycle.md` (deferred authoring per D-GRAD; see backlog idea) and may be invoked by operator at any vault under their authority.
- **Framework-side cycles RUN at III.aDNA** — measure canonical-framework dimensions (`cross_vault_relationship_fidelity` + `vault_card_completeness` + `registry_freshness` operate against the canonical Mermaid edges + frontmatter parsers + sync-drift detection that III.aDNA Argus owns). Framework-side cycles are dispatched via III.aDNA Argus authority per ADR-025 Clause B.
- **Coordination contract**: both sides share dimension definitions (single source of truth per Clause B); both sides emit coord memos at cycle close (per Clause C ack-debt-protocol); both sides commit results to a federation-discoverable persistence layout (M3.7 design spec §2 P6 names `aDNA.aDNA/what/measurement/iii_results/`; III.aDNA canonical-side may mirror at `III.aDNA/what/measurement/iii_results/` or persist directly to a federation-accessible path).

**Acceptance test**: for any proposed III-decadal cycle, the cycle author MUST be able to name (a) which vault hosts the cycle execution; (b) which dimensions the cycle measures from the 5 T12 dimensions; (c) which downstream vaults receive coord memo at cycle close per Clause C. If any of (a)-(c) is ambiguous, the cycle is REJECTED under this Clause A and re-scoped via operator AskUserQuestion.

**Sub-clause A.1 — Cycle-host derivation**: dimension-to-host mapping (canonical at ratification; updates via ADR-025 amendment discipline per Sub-clause C.3):

| Dimension | Cycle host | Rationale |
|-----------|------------|-----------|
| `persona_growth_v0` | aDNA.aDNA | measures operator-vault persona blurbs; operator-side authority |
| `research_context_generation_v0` | aDNA.aDNA | measures adopter-scenario research-context generation; operator-side authority |
| `cross_vault_relationship_fidelity` | III.aDNA | measures Mermaid-edge ↔ MANIFEST.md dependency parity; canonical-side authority |
| `vault_card_completeness` | III.aDNA | measures frontmatter + body section + cross-reference completeness; canonical-side authority |
| `registry_freshness` | III.aDNA | measures sync-drift between vault_card `last_synced` and `inventory_vaults.yaml` canonical; canonical-side authority |

**Sub-clause A.2 — Cycle-host override for offline/degraded operation**: when III.aDNA is unreachable (network partition; III.aDNA offline; III.aDNA advisory-mode-only per ADR-026), framework-side dimensions MAY temporarily downgrade to aDNA.aDNA-side cycle execution using the M3.6 P3 ledger-observation polling skeleton (Appendix A) as fallback per ADR-026 Clause B. Override is operator-discretionary + flag-documented in the cycle result.

### Clause B — Dimension-definition-ownership: III.aDNA canonical authority

**III.aDNA is the canonical authority for dimension definitions across the lattice**. Specifically:

- **Each of the 5 T12 dimensions has a canonical definition at `III.aDNA/what/measurement/dimensions/<dim>_v0.md`** (file authoring DEFERRED to III.aDNA Campaign E; M3.7 spec does not author at III.aDNA per hard constraint #11 zero-III-touches).
- **Each canonical definition specifies**: (a) dimension name + `_v0` suffix; (b) measurement-stub semantics (input artifacts, computation algorithm, output range); (c) measurement-module reference (algorithm implementation path); (d) update + supersession path (`_v0` → `_v1` via III.aDNA Campaign E or successor ratification).
- **Authority boundary**: aDNA.aDNA + consumer wrappers + LatticeAgent.aDNA + Platform.aDNA consumers consume the dimension definitions read-only via federation channels. Authoring authority STAYS at III.aDNA Argus.

**Sub-clause B.1 — Dimension supersession**: when a `_v0` dimension supersedes to `_v1` (e.g., `persona_growth_v0` → `persona_growth_v1`), III.aDNA Campaign E (or successor) authors the new dimension definition + publishes coord memo per Clause C ack-debt-protocol. Existing `_v0` cycle results stay at vault_card `iii_target.<dim>_v0.*` fields per M3.7 P2 schema (additive `_v1` field added; no deletion of `_v0` per M3.7 P6 retention policy). vault_card schema accommodates per-vault-card opt-in cascade for `_v1` adoption.

**Sub-clause B.2 — Consumer-side dimension extensions**: if a consumer vault (e.g., RareHarness.aDNA + Asclepius; CakeHealth.aDNA + Berthier) needs a domain-specific dimension beyond the canonical 5, the consumer MAY author a vault-side dimension at `<consumer-vault>/what/measurement/dimensions/<dim>_v0_<vault_namespace>.md` (e.g., `clinical_safety_v0_rareharness.md`). Consumer-side dimensions are vault-local only — they do NOT federate cross-vault unless promoted via III.aDNA Campaign E + Argus authority. Naming convention: `<dim>_v0_<vault_namespace>` (suffix disambiguates from canonical).

**Sub-clause B.3 — III.aDNA Campaign E entry conditions**: III.aDNA Campaign E (or named successor) opens at III.aDNA-operator-discretionary timing post-M3.7 close. Entry conditions: (a) ADR-025 ratified at M3.7 (satisfied at this S2 ratify); (b) M3.7 design spec ratified (satisfied at this S2 close); (c) III.aDNA Argus authorizes Campaign E open (III.aDNA-operator authority).

### Clause C — Ack-debt-protocol coord-memo cycle

**III-decadal cycle results emit coord memos** to relevant downstream consumer vaults at cycle close. Coord-memo discipline:

- **Coord memo path**: `<vault>/who/coordination/coord_iii_decadal_<source_vault_id>_<cycle_n>_<YYYY-MM-DD>.md` — written at recipient vault by source vault's cycle-orchestration skill.
- **Ack-window**: ≤ 24 hours for non-urgent (default) / ≤ 4 hours for load-bearing (cycle result triggers downstream actions like remediation dispatch; cycle author flags load-bearing in memo frontmatter).
- **Ack mechanism**: recipient persona acknowledges by either (a) updating recipient vault STATE.md with cycle-result reference + acknowledgment; OR (b) replying via coord memo at source vault `who/coordination/coord_iii_decadal_<source_vault_id>_<cycle_n>_ack_<recipient_vault_id>.md`; OR (c) implicit ack via downstream action that consumes the result (logged in source vault STATE.md as observed ack).
- **Ack-debt accumulation**: if ack-window elapses without ack, source vault flags the recipient with `ack_debt: open` in source vault's coord-memo index. Ack-debt does NOT block subsequent cycles, but accumulates as cross-vault coordination signal for retrospective.

**Sub-clause C.1 — Coord-memo content scope**: each coord memo includes (a) cycle ID + source vault + cycle dimensions; (b) result summary (per-dimension score; trend vs. prior cycle if applicable); (c) recipient-specific implications (e.g., "your vault's `cross_vault_relationship_fidelity` score dropped 0.3 since last cycle; investigate Mermaid edge drift"); (d) ack window (default 24h; load-bearing 4h); (e) recipient action requested (if any); (f) source vault contact (persona + agent).

**Sub-clause C.2 — Privacy-preserving aggregation**: coord memos may include score + last_measured + notes per dimension (vault-readable to recipient). Cross-vault aggregations (per M3.7 P5 wrapper airlock-activation audit aggregation + M3.7 P6 result persistence `_aggregations/` layout) include score + last_measured only (no notes; preserves operator-visible commentary at vault-local scope per M3.7 P2 + P5 privacy contract). The `notes` field stays vault-local.

**Sub-clause C.3 — Amendment discipline**: ADR-025 amendments append rows to a per-amendment table documenting (a) the amendment trigger (operator decision / III.aDNA Campaign E v1 promotion / new dimension addition / sub-clause refinement); (b) the amendment date; (c) the amendment author (agent + operator); (d) the amendment scope. Amendments preserve the distributed-but-coordinated invariant per Clause A. Amendments do NOT supersede ADR-025 unless a follow-up ADR explicitly does so (e.g., a future ADR may supersede if III.aDNA Campaign E promotes the entire ADR-025 surface into a III canonical artifact).

## Consequences

### Positive

- **Cross-vault III-decadal coordination becomes operationable**: the 5 T12 dimensions get clear hosting + authoring + propagation contracts at this ratification. v8 P5 100-III-loops capstone opens with a known cycle-location-shape; III.aDNA Campaign E opens with a known dimension-definition-ownership authority; cross-vault coord memos flow under a known ack-debt-protocol.
- **Distributed-but-coordinated avoids over-centralization risks**: framework-side dimensions stay at III.aDNA (canonical authority; avoids fragmentation); operator-side dimensions stay at aDNA.aDNA (operator-vault-context authority; avoids III.aDNA carrying operator-vault-personalization burden). Each side owns what it's best-positioned to own.
- **`_v0` suffix preserved across canonical + consumer extensions**: dimension supersession (Sub-clause B.1) + consumer-side dimension extensions (Sub-clause B.2) both honor the `_v0` discipline introduced at M3.5 T12 spec. Future maturation does not break current consumers.
- **Ack-debt-protocol enables cycle volume without operator-gate explosion**: Campaign SO #1 phase-gates-are-human-gates stays operable under high cycle volume because cycles between cycles only require ack-debt accumulation, not operator gates.
- **4th Campaign SO #14 in-phase exception clause invocation reinforces graduated variant**: post-graduation reinforcement strengthens the canonical-for-load-bearing-decisions-that-block-cross-vault-propagation pattern. `skill_in_phase_adr_ratification.md` post-graduation 4/3+.
- **Substrate-inversion-with-ADR variant GRADUATES at 3rd canonical instance**: M3.4 1st + M3.6 2nd + M3.7 3rd; ≥ 3 instances rubric satisfied; `skill_substrate_inversion_with_adr.md` graduation candidate ADVANCES 2 → 3 of 3 = GRADUATES.

### Negative

- **Per-dimension cycle-host derivation overhead**: each new dimension added at v8 P5 or later requires Sub-clause A.1 amendment (cycle-host mapping). Mitigation: A.1 table is concise; ADR-025 amendment discipline (Sub-clause C.3) is lightweight.
- **Coord-memo volume scales with cycle count × downstream consumers**: high cycle volume produces many coord memos; recipient vaults may struggle with ack-window adherence. Mitigation: Sub-clause C.1 scopes coord memo content to relevant recipient-specific implications (no broadcast); recipients can downgrade to ack-debt accumulation per Clause C without breaking cycle scheduling.
- **III.aDNA Campaign E timing is III.aDNA-operator-discretionary**: this ADR does NOT force III.aDNA Campaign E to open at any specific time. If III.aDNA Campaign E delays indefinitely, the dimension definitions remain unauthored (placeholder forward-stubs only) + measurement modules remain unimplemented + III-decadal cycles cannot actually run. Mitigation: Clause A Sub-clause A.2 offline-fallback provides aDNA.aDNA-side cycle execution via M3.6 P3 + ADR-026 Clause B; v8 P5 100-III-loops capstone may proceed with partial dimension coverage if III.aDNA Campaign E lags.
- **Coordination contract amendment overhead**: high amendment frequency would degrade ADR-025 readability. Mitigation: amendments expected to be infrequent (dimension supersession every 6-12 months at most; cycle-host mapping changes rare).

### Neutral

- **No new skills authored at M3.7**: skill file authoring for `skill_iii_modular_cycle.md` + `skill_decadal_aar.md` updates deferred per D-GRAD=ratify-as-finding-defer-skill-authoring inherited from M3.4 + M3.5 + M3.6 precedent. Backlog idea file extends with M3.7 entries at close cascade.
- **No new ADRs for individual dimensions**: the 5 T12 dimensions all live under ADR-025 + ADR-026 umbrella. Individual dimensions that need their own ADR (e.g., if `clinical_safety_v0_rareharness` grows into a canonical 6th dimension) are routed via III.aDNA Campaign E + Argus authority per Sub-clause B.2 promotion path.
- **Cycle volume forecast neutral at v8 P5 capstone scale**: 100-III-loops capstone runs ~10 cycles per dimension across 31 vault_cards = ~50 framework-side + ~50 operator-side cycles + ~100 coord memos at acquaint cadence. Ack-debt-protocol designed for this scale.

## Alternatives considered

### Alternative 1 — No ADR; III-decadal coordination as design-spec text only (Path B)

Considered at M3.7 S2-entry plan G2 AskUserQuestion. Operator chose Path A. **Why not B**: without ADR-025, the cycle-location-shape + dimension-definition-ownership + ack-debt-protocol would live as design-spec prose; III.aDNA Campaign E + v8 P5 + cross-vault consumers would have no canonical doctrinal anchor; cross-vault coordination would lack a predictable channel. Also forecloses 4th Campaign SO #14 invocation post-graduation reinforcement + substrate-inversion-with-ADR variant 3rd instance graduation.

### Alternative 2 — Centralize III-decadal cycles at III.aDNA only (NO distributed shape)

Considered at design phase. **Why rejected**: III.aDNA would carry operator-vault-personalization burden (persona blurb drift detection requires operator-vault-context); the canonical-side authority would conflate framework-quality measurement with operator-specific measurement. Distributed-but-coordinated (Clause A) preserves the separation: III.aDNA owns canonical-framework dimensions; aDNA.aDNA owns operator-side dimensions; both coordinate via shared definitions + coord-memo discipline.

### Alternative 3 — Centralize III-decadal cycles at aDNA.aDNA only (NO III.aDNA-side cycles)

Considered at design phase. **Why rejected**: aDNA.aDNA would re-derive dimension definitions per cycle (fragmentation risk); III.aDNA's role as canonical framework would shrink to documentation-only (loses operational authority); cross-vault federation breaks (other consumer vaults could not consume canonical dimension definitions if not from a single source). Distributed-but-coordinated (Clause A) preserves III.aDNA's canonical role + enables operator-side cycle execution.

### Alternative 4 — Split ADR-025 into 3 ADRs (Clause A + B + C as separate ADRs)

Considered at design phase. **Why rejected**: the 3 clauses compose cleanly as one ADR; splitting into 3 ADRs would fragment the canonical doctrine surface + increase coord overhead per consumer. The 3-clause ADR is a stronger pattern than 3 separate ADRs (mirrors ADR-024 Clause A + B + C composition pattern at M3.6).

### Alternative 5 — Defer ADR-025 to v8 P5 ratification (path B variant)

Considered implicitly via design-at-P3-propagation-at-P5/P6 doctrine. **Why not deferred**: v8 P5 opens at operator-decisioned P3 → P4 phase-exit gate; without ADR-025 in hand at v8 P5 entry, v8 P5 might re-derive coordination contract OR delay first cycle until ADR-025 ratifies. M3.7 ratifying ADR-025 now means v8 P5 can open with a ready coordination contract + dispatch first cycle without re-deliberation. Also forecloses 4th Campaign SO #14 invocation reinforcement opportunity.
