### Successful patterns

1. **3-session shape generalizes to a third independent instance**: M08a authoring (S1 spec + bulk render / S2 finalization + announcement drafts / S3 mission close) + M03 implementation (S1 spec + ADR draft / S2 destructive flatten + V/R harness / S3 close + AAR + ADR ratification) + **M04 implementation** (S1 spec / S2 destructive bootstrap + workspace router + template hook + scaffolds + skills + backlog ideas / S3 audit + AAR + mission close). The shape's invariants: (i) S1 is non-destructive — spec authoring + optional ADR drafting; (ii) S2 holds the destructive ops + the main work product; (iii) S3 is mechanical close — audit/verification artifact + AAR + status flips. Variations are confined to S2 content (different per mission class + mission specifics). Strong evidence the shape is mission-class-agnostic across the two classes that have produced it (authoring + implementation). Next instance candidate: M05 publish-skill rewrite (implementation; expected to fit 3-session shape).
2. **End-to-end fork-and-customize production validation**: M03 R1-R11 regression test (S2 in `/tmp/sandbox_v7/`) demonstrated `skill_project_fork.md` could fork a fresh template + apply exclusion rules R2-R7 + pass naming validation. Future vault-bootstrap missions can rely on M04 as the canonical production-reference (M03 was sandbox-only; M04 is production).
3. **Inline persona installation pattern confirmed**: Hestia installed verbatim from M01 Obj 3 §3 into `node.aDNA/CLAUDE.md` lines 18-61 (Identity & Personality section). The pattern works across persona categories (project, framework, platform, org-vault, **per-node operational** as M04 introduces).

Plan-mode approval at S2 entry surfaced all 5 in a single AskUserQuestion block; all 5 landed with defaults without revision. The pattern (spec lists decisions + defaults; plan-mode surfaces them; operator approves or overrides) is now canonical for multi-decision missions. Compared to M03 which had 4 decisions (also all defaulted without revision), M04 confirms the pattern scales — 5 decisions are not more friction than 4 when the spec surfaces them up-front.

The discipline surfaces template-level gaps for systematic resolution.

### Surprises and friction

1. **`inventory_memberships.yaml` federation block has 8 keys, not 5**: M01 Obj 3 §8 D7 prose enumerated 5 federation keys (`shareable`, `discoverable`, `source_instance`, `version_policy`, `share_policy`) with the FAIR fields (license, creators, keywords) implicitly carried over from MANIFEST.md. M04 S2 implementation landed all 8 keys directly in the federation block, and `skill_node_health_check.md` Step 8 documents the 8-key requirement explicitly. The result is more self-contained (federation block is fully readable without cross-referencing MANIFEST FAIR), but constitutes a minor design-doc drift. No score impact (D7 = 3 either way; federation block is intentionally inert by default). **Lesson**: when a design artifact enumerates a list of required keys for a structural block, the implementation may legitimately add convenience-copies of related metadata; surface this in the implementation's verifier (skill_node_health_check.md Step 8) so future audits don't read the original design as canonical. Backlog-idea-worthy: tighten M01 Obj 3 §8 D7 prose to enumerate all 8 keys (M07 absorb or v8.0+ ontology refinement).
2. **`who/coordination/registry_stub.md` prescribed by M01 Obj 3 §8 D8, not produced at M04 S2**: The M01 Obj 3 §8 D8 row's "What M04 must produce to hit the score" column prescribed "M04 produces a registry-stub at `who/coordination/registry_stub.md`" — but the M04 spec's Deliverables table didn't include it as a numbered deliverable (the deliverables table covered scaffolds + skills + workspace router + backlog ideas + audit + AAR, but not the registry stub). At S2 the stub wasn't produced; at S3 audit the score landed at the predicted 3 anyway because the registration-availability framing scores 3 regardless of stub presence. **Lesson**: when a design artifact's per-dim rationale prescribes a specific document and the corresponding implementation mission's deliverables table doesn't include it, the audit should still score correctly if the underlying capability is inherited (registration via `skill_lattice_publish.md` inheritance covers D8). Backlog-idea-worthy: when M07 absorbs vault-registration concept extension, produce `registry_stub.md` then (or close the design-doc gap by removing the prescription).
3. **README.md no-frontmatter convention check during audit**: At audit time, the `grep -c "^---$"` on `node.aDNA/README.md` returned 0 (no frontmatter delimiters). Was this a D3 frontmatter gap? Verified against `.adna/README.md` (template source) + `aDNA.aDNA/README.md` (sibling Org-Vault, the campaign-hosting vault itself) — both ALSO have no opening frontmatter (the `---` matches found are body HR separators). Confirmed README.md is the GitHub-landing-page-style human-facing entry, not an indexed content file; intentionally unfrontmatted per ecosystem convention. **Lesson**: when auditing frontmatter compliance, README.md should be excluded from the "all content files require frontmatter" rule. Backlog-idea-worthy: tighten `aDNA.aDNA/CLAUDE.md §Compliance Dimensions D3` to explicitly list README.md as an exception (or, alternately, define "content files" narrowly enough that README.md is naturally excluded).
4. **The 12-step `skill_node_health_check.md` validator was not actually run during S3 audit**: M04 spec §Acceptance criteria #11 states "`node.aDNA/` `skill_node_health_check.md` exits with code 0 (no drift detected post-bootstrap)". The skill is documented as a 12-step validator with exit codes 0-11. At S3 audit, the 12-step procedure was *spot-walked* by the agent (filesystem reads + grep + structural verification) rather than *executed* as a single procedural run. Result was equivalent (all 22 deliverables present, frontmatter valid, federation block valid, etc.) but the acceptance criterion's exit-code-0 claim is technically aspirational unless the skill is implemented as an executable script. **Lesson**: when a skill is authored as a 12-step validator with exit codes, decide at authoring time whether the agent or a wrapping bash script is the canonical executor. If agent-executed: the acceptance criterion should read "agent confirms each step PASS" rather than "exits with code 0". If script-wrapped: provide the wrapping script. Backlog-idea-worthy: M07 absorb either by adding a bash wrapper to skill_node_health_check.md OR by softening the acceptance criterion's exit-code language to agent-walkthrough language.
