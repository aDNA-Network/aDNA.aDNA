## §6 Self-reference note (Standing Order #2)

This deliverable demonstrates the standard's evolution **using** the standard, in three concrete ways:

1. **The mission file (`mission_adna_infra_planning_01.md`) IS a working example of the `mission` entity-type** — it carries the v6.0 frontmatter (`type: mission`, `mission_id`, `campaign`, `campaign_phase`, `status`, `mission_class`), uses the AAR-mandate pattern (Standing Order #5), embeds 11 objectives that consume Stage 1 refinement output, and follows the per-objective layout that `template_campaign_mission.md` codifies. The file describing how to evolve aDNA *is* aDNA.

2. **The Skill inventory discrepancy (Issue I-2) is itself a self-reference finding** — the Skills inventory table inside `.adna/CLAUDE.md` (the canonical source of skill names) drifted from `.adna/how/skills/*.md` (the canonical source of skill files). This drift is invisible to anyone reading just the table; it surfaces only when an agent counts files. The standard's own governance file is failing the dual-audience test on this single row. M07 fixes the table; this Obj-1 deliverable is the artifact that recorded the drift.

3. **The campaign trigger imprecision (Issue I-6) is a self-reference finding about how aDNA refines itself** — Stanley's original framing ("skill_lattice_publish uses a .publish-clone/ workaround") was *operationally* correct (the workaround exists; vaults lack remotes; publishing is broken) but *file-locally* imprecise (the skill file itself doesn't have rsync code; the workaround is ambient). The Stage 1 + Stage 2 work refined the imprecision into a precise mission scope (M05 creates new skills; doesn't rewrite the registry one) — and *that* refinement is recorded here. The pattern is: rough operator intent → planning-mission refinement → ADR-codified scope. The campaign demonstrates the planning protocol it uses.

These three threads are why ADR-004 keeps this campaign in `aDNA.aDNA/` rather than migrating it to `node.aDNA/` after M04: the campaign IS a working example of the standard, governed by the standard, evolving the standard. Keeping it where the standard's documentation lives keeps the demonstration visible.
