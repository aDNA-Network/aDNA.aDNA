## Findings (extended — 3-session implementation-mission scope)

### Successful patterns

1. **3-session decomposition for implementation-class missions**: S1 non-destructive spec + ADR-008 draft (operator-approval boundary at S1→S2) / S2 destructive execution (Stages A–D per runbook §8) + V/R harness 24/24 PASS / S3 mechanical close + AAR + ADR-008 ratification. The shape mirrors [[aar_m08a_upgrade_guide_and_coord_memos.md|M08a's 3-session authoring-mission arc]] (Session 1 spec + Obj 1 bulk render / Session 2 finalization + announcement drafts / Session 3 mission close). The pattern is **mission-class-agnostic** — buffers against context exhaustion AND surfaces an operator-approval boundary at the right moment (Standing Order #1: phase gates are human gates). See [[#Conceptual contributions]] #2 for the generalization.
2. **Pre-flight collision-resolution discipline (B0 + B1 micro-sequence)**: Two name collisions were known to potentially block the B6 flatten promotion — (i) outer `adna/CLAUDE.md` (5.8K workspace router) vs inner `.adna/CLAUDE.md` (21K template governance) when both promoted to root; (ii) outer `adna/README.md` (5.9K GitHub landing) vs inner `.adna/README.md` (47K aDNA spec) at the same level. Collision (i) was identified in [[../mission_adna_infra_p1_03_repo_flatten.md|M03 spec §Risks R1]] before S2 and resolved by B1 (outer wrapper → `.adna/how/templates/template_workspace_claude.md` per ADR-007 §Decision; moves outer wrapper into inner tree before promotion). Collision (ii) was discovered by the S2-entry pre-flight Explore agent (not flagged in S1) and surfaced as Operator Decision 4 (preserve both; outer → root; inner → `.adna/what/docs/aDNA_overview.md`); resolved by B0 before B6 ran. Both collisions handled cleanly. **Lesson**: when a flatten/promotion will potentially collide, pre-resolve in dedicated sub-commits that move colliding items out of the collision path, then run the promotion.
3. **CHANGELOG verbatim-copy discipline**: [[m01_obj6_version_bump_checklist.md|M01 Obj 6 §1]] pre-drafted the v7.0 CHANGELOG entry during M01 Stage 2 Session 4 (2026-05-08). M03 spec §Hard constraints included "CHANGELOG verbatim-copy not re-authoring". B10 copied the entry without modification (only a forward-looking metadata note prepended explaining M03-vs-M06 attribution + bullet status). M06 fills in date placeholder + verifies forward-looking bullets at tag execution. **Lesson**: separating pre-drafting (planning mission) from copying (execution mission) prevents drift; the discipline scales to other "pre-drafted content + later execution" patterns.
4. **Risk 3 V12/V13 syntax-validation fallback**: V12/V13 are interactive agent-facing checks (open `claude` from `~/aDNA/` + observe greeting). An in-session agent inside `aDNA.aDNA/` can't fully test these by launching a new agent. The S1 review process (independent Explore-agent critique + Plan-agent validation) added a Risk 3 mitigation to the spec at refinement time: validate frontmatter YAML syntax via `python3 yaml.safe_load` as the structural-integrity proxy. Both `~/aDNA/CLAUDE.md` (workspace router) and `~/aDNA/.adna/CLAUDE.md` (template governance) parsed cleanly at S2 Phase D; operator-verifies actual greeting at next fresh `claude` session. The mitigation distinguishes **structural errors** (frontmatter break, broken wikilinks — rollback-worthy) from **agent-context errors** (persona mismatch, missing content — in-place fix). **Lesson**: when a verification check is interactive + non-self-testable, surface a programmatic fallback that distinguishes structural-failure from runtime-failure.
5. **Per-bucket operator-decision surfacing**: 4 decisions surfaced explicitly via AskUserQuestion + documented in [[../mission_adna_infra_p1_03_repo_flatten.md|M03 spec §Operator decisions required at Session 2 entry]] + recorded in S2 SITREP: (D1) `prepare_for_onboarding.sh` directory-move vs rename → Option A directory-move; (D2) node.aDNA workspace-router Step 0 block include S2 vs defer to M04 → Option B defer to M04; (D3) Compute Tiers/link-table trim vs retain → Option B retain; (D4) README collision preserve-both vs single-survivor → preserve both. The pattern keeps operator-discretionary calls **visible + auditable** rather than buried in agent default behavior. Defaults documented for fallback ("if operator hasn't specified by Pre-flight close, default to A/B/B"). **Lesson**: when a mission has 2+ discretionary calls, build an §Operator decisions section into the spec + surface as a single AskUserQuestion block at entry; defaults handle absent-operator cases.
6. **Separate aDNA.aDNA commit from upstream LatticeProtocol/adna commits**: M03 S2 produced 14 upstream commits (B0-B10 + C1-C3; pushed to `LatticeProtocol/adna`) + 1 aDNA.aDNA commit (`c3f184c` in this campaign vault closing S2). Clean attribution: each repo's history reflects its own scope — template structure changes upstream; campaign governance + session records + V/R artifacts in aDNA.aDNA. S1 was 1 commit each (different repos); S3 (this close) will be 1 commit each. **Lesson**: when a mission spans both a campaign vault and an external repo, push commits to each repo independently under role-appropriate commit messages; don't conflate.

### Surprises and friction

1. **B5 `.gitignore` amendment caught accidentally-staged `.adna/.obsidian/workspace.json`**: Initial B5 commit rewrote the template-level `.gitignore` for the post-flatten flat-repo structure (paths-relative-to-repo-root). The rewrite removed the pre-flatten path entry `.adna/.obsidian/workspace.json` (since post-flatten the file would be at `.obsidian/workspace.json`). But at B5 commit time, the flatten promotion (B6) had NOT yet run — the file was still at `.adna/.obsidian/workspace.json`. Removing the entry exposed the previously-ignored file to `git add -A`, staging it for commit. Diagnosed pre-commit via diff review (the commit showed an unexpected `create mode 100644 .adna/.obsidian/workspace.json`). Amended commit: `git rm --cached` the file + updated `.gitignore` to keep BOTH pre-flatten and post-flatten path forms during the transition (post-B6 the pre-flatten entries become harmless no-ops but cause no harm). **Lesson**: when rewriting `.gitignore` mid-restructure (paths-in-flight), keep both pre-state and post-state path forms; remove the pre-state forms in a follow-up commit after the flatten promotion lands.
2. **Agentic-DNA sweep target undercount in S1 Phase A spot-check**: Session 1 Phase A spot-check claimed "1 reference in .adna/CLAUDE.md"; actual grep at B8 found ~30 references across 19 files. Caught by the S1 review process (independent Explore-agent critique surfaced the discrepancy via Fix A in the spec refinement). M03 spec Obj 3 Commit 8 was **functionally correct** ("sweep all hits" wildcard coverage handled the actual count), but the prose claim was cosmetically inaccurate. **Lesson**: when claiming a sweep-target count, run the sweep query against ALL in-scope files (not just a subset spot-check) and record the actual count. The "sweep all hits" wildcard saved the day this time; relying on a narrow spot-check is brittle.
3. **GitHub canonical-case divergence**: The S2 push response surfaced "This repository moved. Please use the new location: https://github.com/LatticeProtocol/aDNA.git" — GitHub canonicalized the renamed repo to `LatticeProtocol/aDNA` (mixed case per `.aDNA` suffix convention) rather than ADR-006's strict-lowercase `LatticeProtocol/adna`. The local `LatticeProtocol/adna.git` URL works via GitHub's case-insensitive routing; the push succeeded. Cosmetic divergence between ADR-006 spec text and external state. **Lesson**: when an ADR specifies a string-literal external identifier (URL slug, repo name, hostname), verify the actual external state matches the spec at execution time and reconcile divergences in the AAR. ADR-006 amendment OR M07 cleanup will resolve.
4. **M03 spec §6 reference typo (should be §1)**: M03 mission spec Obj 3 Commit 11 referenced "CHANGELOG verbatim-copy from M01 Obj 6 §6" — but the pre-drafted CHANGELOG entry is at §1 of `m01_obj6_version_bump_checklist.md` (§6 is the GitHub release notes template, which is different). Caught at B10 execution by reading the source artifact directly. Cosmetic typo in the M03 spec; corrected via B10's commit message ("note §6 reference is a typo for §1"). **Lesson**: when authoring spec cross-references to a multi-section input artifact, verify the section number against the actual artifact, not against memory of section numbers.

### Conceptual contributions

1. **ADR-008 is the 5th instance in the airlock lineage** (load-bearing finding — see §AAR lightweight Finding above + §Items deferred forward-references): III canonical (1 spec) → VideoForge reference (1 instance) → CanvasForge worked example (1 bilateral commission) → M08a multilateral (17 parallel relationships) → **template stub** (every forked vault inherits the entry point structurally; activation is content-only). The lineage is now a **5-element series** that the v3-EC `M05-EC` mission will instantiate into 19+ per-vault airlock instances when the per-vault rollout campaign opens.
2. **3-session shape is mission-class-agnostic**: M08a (authoring-class) and M03 (implementation-class) both fit the same 3-session decomposition pattern (work + harness/verification + close). Strong evidence that the shape **generalizes across mission classes** when the work is large enough to require multiple sessions but small enough to bound at 3. Future verification-class missions (M08b post-flatten propagation receipts) may exhibit a different shape (verification can collapse to 1 session per M02 single-execution precedent — see [[aar_m02_ecosystem_matrix.md|M02 AAR]]). Future destructive-class missions inherit this 6-layer pattern as a reusable template — see §Successful patterns #2 for the pre-flight collision-resolution component.
